---
title: "Upsert"
description: "Insert or update entities in a Milvus collection. Existing entities are overwritten; new entities are inserted."
---

The `upsert` method combines insert and update semantics. When an entity with a matching primary key already exists in the collection, it is overwritten with the new data. When no match is found, a new entity is inserted. Upsert behaves atomically per entity.

<Warning>
  Upsert requires the collection to have `auto_id` **disabled**. If `auto_id` is enabled, primary keys are generated by the server and cannot be referenced for updates — use `insert` instead.
</Warning>

## Parameters

<ParamField path="collection_name" type="string" required>
  The name of the target collection.
</ParamField>

<ParamField path="data" type="list[dict]" required>
  A list of entities to upsert. Each entity is a dictionary mapping field names to values. The primary key field must be included to determine whether to insert or overwrite.

  <Expandable title="Entity structure">
    Each dictionary must contain the primary key field and all other non-nullable fields:

    ```python
    {
        "id": 42,
        "text": "Updated description",
        "vector": [0.7, 0.1, 0.5, ...]
    }
    ```

    If an entity with `id: 42` already exists, it is replaced in full. If it does not exist, it is inserted as a new entity.
  </Expandable>
</ParamField>

<ParamField path="partition_name" type="string">
  The name of the target partition. If omitted, the default partition (`_default`) is used.
</ParamField>

## Response

<ResponseField name="upsert_count" type="integer">
  The total number of entities processed (inserts plus updates combined).
</ResponseField>

<ResponseField name="insert_count" type="integer">
  The number of entities that were newly inserted (no prior matching primary key).
</ResponseField>

<ResponseField name="delete_count" type="integer">
  The number of existing entities that were replaced (prior matching primary key found and tombstoned).
</ResponseField>

<ResponseField name="ids" type="list[int] | list[str]">
  The primary key values of all upserted entities.
</ResponseField>

<ResponseField name="timestamp" type="integer">
  The server-side hybrid logical timestamp at which the upsert was acknowledged.
</ResponseField>

## Example

<CodeGroup>

```python Basic upsert
from pymilvus import MilvusClient

client = MilvusClient(uri="http://localhost:19530")

# Entity with id=1 will be overwritten if it exists; otherwise inserted
data = [
    {"id": 1, "text": "Revised description for entity 1", "vector": [0.9, 0.1, 0.2, 0.4, 0.6]},
    {"id": 4, "text": "Brand new entity",                  "vector": [0.2, 0.8, 0.5, 0.1, 0.3]},
]

res = client.upsert(
    collection_name="my_collection",
    data=data,
)

print(res)
# {
#   "upsert_count": 2,
#   "insert_count": 1,
#   "delete_count": 1,
#   "ids": [1, 4]
# }
```

```python Upsert into a partition
from pymilvus import MilvusClient

client = MilvusClient(uri="http://localhost:19530")

data = [
    {"id": 100, "category": "tech", "vector": [0.5, 0.5, 0.5, 0.5, 0.5]},
]

res = client.upsert(
    collection_name="my_collection",
    data=data,
    partition_name="tech_partition",
)

print(f"Upserted {res['upsert_count']} entity")
```

</CodeGroup>

## Notes

- **How upsert works internally**: An upsert is implemented as a delete of the existing entity (if found) followed by an insert of the new entity. The `delete_count` in the response reflects the number of entities that were replaced.
- **auto_id restriction**: Collections with `auto_id=True` cannot use upsert because Milvus generates primary keys on insert, making it impossible to reference existing entities by key.
- **Visibility**: Like `insert`, upserted data is buffered and not immediately visible in search results. It becomes searchable after a flush or segment seal.
- **Partial updates are not supported**: Upsert replaces the entire entity. To update a single field, you must supply all field values in the `data` payload.
